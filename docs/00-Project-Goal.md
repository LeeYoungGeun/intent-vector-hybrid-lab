# 00. 프로젝트 목적과 범위

## 목적
이 템플릿은 **“의도 기반 벡터/하이브리드 검색”** 파이프라인을 구성하는 핵심 5요소를, **동일한 테스트셋/동일한 지표/동일한 로그 포맷**으로 비교·검증하기 위한 **API 테스트 전용 공용 프로젝트**입니다.

테스트 대상(5요소)
1. **임베딩(Embedding)**
2. **ANN/VectorDB 리트리벌(Vector Retrieval)**
3. **BM25 + 하이브리드 퓨전(Hybrid Fusion)**
4. **리랭킹(Re-ranking)**
5. **점수/규칙 기반 필터링(Score/Rule Filtering)**

## 왜 공용 템플릿이 필요한가
- 벤더/모델 교체가 잦아도 **테스트 방식이 흔들리지** 않도록
- “좋아졌다”를 말이 아니라 **숫자(지표)로 고정**하기 위해
- 팀원이 바뀌어도 **동일한 입력/출력/로그 규격**으로 재현 가능하게 하기 위해

## 범위(Scope)
### 포함
- 벤더별 API 호출을 **표준 인터페이스**로 추상화(Adapters)
- 테스트 데이터/정답/판정 룰/스코어 임계값을 **파일 기반**으로 관리
- 성능 측정:
  - 검색 품질(Precision@K, Recall@K, MRR/nDCG 등)
  - 애매함 처리 지표(clarify rate, fail-safe rate)
  - 지연시간(latency), 처리량(QPS), 비용(토큰/호출 수)
- 결과 리포트(요약표 + 상세 로그)

### 제외(Non-goals)
- 완전한 제품형 검색 서비스(운영용 UI/관리 콘솔)
- 결제/권한/대규모 멀티테넌트 운영
- 본 템플릿 자체가 벤더 성능을 ‘공식’ 비교하는 레이팅

## 성공 기준(Template Definition of Done)
- 동일한 테스트셋에서 벤더/모델/검색엔진을 바꿔도:
  - 실행 커맨드가 동일
  - 결과 JSON 스키마가 동일
  - 리포트 포맷이 동일
- “욕실 매트 vs 요가 매트” 같은 애매한 쿼리에서:
  - 하이브리드 + 리랭킹 + 필터링이 **오탐을 줄이는지** 지표로 확인 가능
